# Контекст проекта для Codex

Этот репозиторий пока является проектной базой для домашней умной теплицы / гроубокса для томатов. Главная цель - постепенно перейти от простого мониторинга влажности почвы на ESP32 к полноценному гроубоксу с датчиками, историей измерений, уведомлениями, автополивом, управлением светом/вентиляцией и сервером на Ktor.

## Текущий состав репозитория

- `smart-greenhouse-plan.md` - главный план реализации: этапы, архитектура, API, схема БД, прошивка ESP32, риски.

## Целевая архитектура

Базовый поток данных:

```text
Датчики -> ESP32 -> Wi-Fi -> Ktor API -> База данных -> Панель мониторинга / уведомления
```

Ключевые части системы:

- ESP32 читает датчики, усредняет измерения и отправляет JSON на сервер.
- Ktor-сервер принимает телеметрию, хранит историю, отдает API для текущих значений, графиков, калибровки и будущих команд.
- База данных: PostgreSQL как основной вариант; H2/SQLite допустимы только для локального прототипа.
- Миграции: Flyway.
- JSON: `kotlinx.serialization`.
- Авторизация устройств: device token в заголовке `X-Device-Token`.
- Связь с ESP32 для MVP: HTTP POST для телеметрии; для команд на старте предпочтителен HTTP polling. MQTT можно добавить позже.

## MVP

Первый рабочий релиз должен быть надежным, а не красивым:

- ESP32 DevKit + 1-2 емкостных датчика влажности почвы v1.2.
- `POST /api/v1/telemetry` принимает показания от устройства.
- Сервер хранит `devices`, `sensors`, `measurements`.
- Есть калибровка сухого/влажного значения датчика.
- Можно получить последние показания и историю.
- Система переживает временную недоступность Wi-Fi, сервера и перезапуск ESP32.

Рекомендуемый формат телеметрии:

```json
{
  "deviceId": "growbox-esp32-001",
  "timestamp": "2026-05-10T12:00:00Z",
  "readings": [
    {
      "sensorId": "soil-1",
      "type": "soil_moisture",
      "raw": 2380,
      "percent": 42.5
    }
  ],
  "batteryVoltage": null,
  "rssi": -61
}
```

## Ориентиры API

MVP endpoints:

- `POST /api/v1/telemetry` - прием показаний от ESP32.
- `GET /api/v1/devices` - список устройств.
- `GET /api/v1/devices/{id}/latest` - последние показания.
- `GET /api/v1/devices/{id}/readings?from=&to=` - история показаний.
- `POST /api/v1/sensors/{id}/calibration` - обновление калибровки.
- `GET /api/v1/health` - healthcheck сервера.

Следующие этапы:

- `GET /api/v1/growboxes/{id}/dashboard`
- `POST /api/v1/growboxes/{id}/events`
- `GET /api/v1/alerts`
- `POST /api/v1/alerts/{id}/ack`
- `POST /api/v1/devices/{id}/commands`
- `GET /api/v1/devices/{id}/commands/pending`
- `POST /api/v1/devices/{id}/commands/{commandId}/result`

## Предпочтения по реализации

- Если создается backend, использовать Kotlin + Ktor и структуру модулей из `smart-greenhouse-plan.md`.
- Начинать с простого API и тестируемой доменной модели, не строить тяжелую платформу заранее.
- Для прошивки ESP32 предусмотреть усреднение 5-10 analog-замеров, retry отправки и диагностические сообщения в Serial.
- Не хардкодить секреты Wi-Fi, device token и URL сервера в публичные файлы. Использовать локальные конфиги/секреты, которые не коммитятся.
- Для новых проектных решений сначала проверять существующие markdown-файлы, чтобы не противоречить уже выбранным этапам.

## Безопасность и важные ограничения

- Не использовать резистивные датчики влажности почвы для постоянной работы: они быстро окисляются.
- Не подключать 220V напрямую к ESP32.
- Для света, вентиляции и сетевых розеток сначала предпочитать таймер-розетки или готовые умные розетки/модули в корпусе.
- Любая силовая часть должна быть в закрытом корпусе, с нормальными клеммниками и разделением 220V от 3.3/5/12V.
- Автополив включать только после накопления истории влажности и понимания порогов для конкретной почвы.
- Для автополива обязательны ограничение времени работы помпы, ручной режим, аварийное отключение и датчик протечки.

## Следующий практический шаг проекта

Если пользователь просит "продолжить проект" без уточнений, наиболее вероятный следующий шаг:

1. Создать минимальный Ktor backend.
2. Добавить `POST /api/v1/telemetry`.
3. Добавить модели `devices`, `sensors`, `measurements`.
4. Подготовить локальный запуск и базовые тесты.
5. Затем отдельно написать MVP-прошивку ESP32 для отправки телеметрии.
